## Context

项目已经明确 P0 方向（CLI 语义统一、统一入口、dry-run 预检），但仍缺少从“可用”到“可维护”的 P1 能力定义。当前实现中，配置来源叠加（默认/用户/文件/env/CLI）对用户不可见，错误信息多为技术性字符串，批处理结果缺乏标准化机器输出，导致排障与自动化集成成本偏高。

本设计聚焦 CLI 层与服务编排层，遵循 KISS：在不重构核心转换器的前提下，通过统一入口和轻量适配器补齐 P0 + P1 的最小可交付闭环。

约束：
- 保持向后兼容，旧参数继续可用并提供弃用提示。
- 不引入新外部依赖，不改造核心转换引擎内部实现。
- 仅在参数解析、配置加载可视化、错误输出和执行报告处做最小增量。

## Goals / Non-Goals

**Goals:**
- 完成 P0：统一 CLI 参数语义、统一 `convert` 最小调度、`--dry-run` 无副作用预检、help/README 对齐。
- 完成 P1 最小闭环：配置生效可观测、错误可操作、结构化执行报告。
- 保持兼容与低迁移成本：旧参数可继续运行，逐步引导用户迁移。
- 保持代码路径简单：单一入口、集中映射、统一输出协议。

**Non-Goals:**
- 不重写 `PdfConverter`/`HtmlConverterService`/`MergerService` 等核心处理器。
- 不引入插件系统、任务队列或复杂 pipeline 抽象。
- 不在本次引入远程输入源、watch 模式、性能重构。

## Decisions

1. **建立“CLI 统一规范层”，将参数兼容和语义映射集中处理**
   - 决策：在 CLI 入口与 handler 之间建立单点映射规则，将旧参数映射为新语义字段，再进入统一执行路径。
   - 原因：减少分支散落，兼容逻辑可控，便于后续下线旧参数。
   - 备选：在各命令处理器内分别兼容（拒绝，重复逻辑、易漂移）。

2. **`convert` 作为统一入口，采用“输入类型识别 + 复用现有处理器”**
   - 决策：`convert` 根据输入文件扩展名/目录扫描结果选择现有处理器执行，不新增复杂中间抽象。
   - 原因：最小改动即可统一体验，符合 KISS。
   - 备选：先构建全新多格式转换 pipeline（拒绝，超出 P0/P1 范围）。

3. **dry-run 与实际执行共享同一“计划生成器”**
   - 决策：先生成执行计划（待处理、跳过原因、输出路径），dry-run 仅打印计划；真实执行复用计划进行处理。
   - 原因：确保 dry-run 与真实行为一致，避免“双实现偏差”。
   - 备选：dry-run 独立实现（拒绝，一致性风险高）。

4. **配置可观测采用“字段来源标注”而非全量 debug 日志**
   - 决策：输出关键配置项的最终值及来源（default/user/file/env/cli），不输出冗长内部细节。
   - 原因：满足排障需要且保持输出简洁。
   - 备选：暴露全部合并过程（拒绝，噪音大且维护成本高）。

5. **错误输出统一协议：错误分类 + 建议动作 + 上下文**
   - 决策：CLI 层统一捕获并格式化为用户可执行提示（例如路径不存在→检查路径/权限）。
   - 原因：降低失败恢复门槛，减少“看不懂错误”。
   - 备选：维持原始异常透出（拒绝，用户体验差）。

6. **结构化报告采用 JSON 文件并保持可选输出**
   - 决策：批处理结束后可输出标准 JSON 报告（总数、成功/失败、失败明细、耗时、配置摘要）。
   - 原因：便于 CI 与自动化消费，同时不影响人类可读终端输出。
   - 备选：仅终端文本摘要（拒绝，机器消费困难）。

## Risks / Trade-offs

- **[风险] 兼容层与新语义并存导致短期复杂度上升** → **缓解**：兼容逻辑集中在单文件，增加弃用提示与测试覆盖。
- **[风险] 输入类型自动识别误判** → **缓解**：优先显式参数，其次扩展名；dry-run 显示判定结果便于提前确认。
- **[风险] 配置来源输出泄露过多信息** → **缓解**：仅输出与执行相关字段，不包含敏感环境变量原值。
- **[取舍] 不重构核心转换器意味着技术债延后** → **缓解**：通过统一入口与协议先稳定外部行为，为后续重构建立边界。
- **[取舍] JSON 报告增加少量维护成本** → **缓解**：采用最小字段集合，避免过度设计。

## Migration Plan

1. 引入参数映射与弃用提示（保持旧参数可运行）。
2. 上线统一计划生成器，并接入 dry-run 与真实执行。
3. 增加配置来源可见输出（默认关闭，按选项/verbose 打开）。
4. 统一错误输出协议并更新帮助文档。
5. 增加结构化报告输出选项与测试覆盖。
6. 发布后观察一版，评估旧参数使用比例，再决定下阶段收敛策略。

回滚策略：
- 若新入口出现兼容问题，可临时回退到旧命令分发路径；
- 保留旧参数映射开关，必要时仅禁用新语义提示，不影响执行。

## Open Questions

- 配置可观测命令形态采用独立子命令（如 `config --print`）还是 `convert --show-config`？
- 结构化报告默认写文件还是仅在指定参数时输出？
- 错误分类编码是否在本阶段公开为稳定契约，还是先内部约定后再对外文档化？
