## Context

当前 CLI 在参数命名与命令入口上存在认知负担：用户需要区分多个命令及含义接近但语义不同的参数，且文档与行为存在不完全对齐。该变更聚焦 P0：在不重构核心转换引擎的前提下，优先提升“可理解、可预期、可安全执行”的命令体验。

约束：
- 遵循 KISS，优先小改动复用现有模块。
- 兼容已有脚本，不做破坏性切换。
- 先保证 CLI 层行为一致，再考虑深层能力扩展。

## Goals / Non-Goals

**Goals:**
- 统一 CLI 参数语义，降低误用概率。
- 让 `convert` 成为最小可用的统一入口（多输入类型统一调度）。
- 提供 `--dry-run` 预检，批处理前可见执行计划。
- 保持向后兼容并给出清晰弃用提示。

**Non-Goals:**
- 不重写转换器内部实现。
- 不引入新的外部依赖或复杂插件体系。
- 不在本次中做大规模性能优化或并发模型重构。

## Decisions

1. **参数语义分离并保留兼容层**
   - 决策：将“输出格式”和“页面规格”在 CLI 语义上明确区分；旧参数继续可用但输出弃用提示。
   - 原因：在不打断现有脚本的情况下快速降低认知错误。
   - 备选：直接移除旧参数（拒绝，迁移成本高、风险大）。

2. **统一入口采用“检测 + 分发”最小模式**
   - 决策：`convert` 在 CLI 层做输入类型识别并分发到现有处理器，避免新增中间抽象层。
   - 原因：复用现有稳定能力，最小改动即可形成统一体验。
   - 备选：先做全新 pipeline 抽象（拒绝，超出 P0/KISS）。

3. **dry-run 仅做预检，不做副作用操作**
   - 决策：`--dry-run` 只输出将处理文件、输出位置、跳过原因，不创建文件、不触发转换。
   - 原因：用户可在零风险下验证参数与路径配置。
   - 备选：dry-run 执行到部分步骤（拒绝，容易引入“看似试跑实则有副作用”）。

4. **文档与帮助信息同版本强绑定更新**
   - 决策：CLI help、README 示例与行为一并修改，保证发布时一致。
   - 原因：P0 目标是降低误用，文档不一致会抵消收益。

## Risks / Trade-offs

- **[风险] 旧参数兼容导致实现分支增加** → **缓解**：兼容层集中在参数解析处，统一映射后进入单一路径。
- **[风险] 自动检测分发在边界输入上误判** → **缓解**：使用明确扩展名/输入类型优先级，并在 dry-run 中可见判定结果。
- **[取舍] 保守改动意味着短期内架构不“最优雅”** → **缓解**：先用最小方案达成可用性目标，后续再按反馈演进。
